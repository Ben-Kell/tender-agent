| Provision of Services (clause 3.1 COD, Attachment A (Services Description and Additional Module Terms)) The Supplier must provide the Services in accordance with the Deed and as further set out in this Work Order. Without limiting any other provision of the Deed, the Supplier must ensure the Services it provides will: meet all specifications, Laws, Authorisations, Performance Measures, Milestones (or if no Milestones are so specified, meet requirements without delay) and timeframes specified in the Deed and this Work Order, including any annexed Plan; be compatible with the Commonwealth’s existing ICT environment including any impacted Existing Systems; be consistent with the Deed Objectives (including the AIC Objectives) and any Work Order Objectives; and minimise any risks or disruptions to the Commonwealth. Background to the Services The Intelligence Capability Division (ICD) within Defence Intelligence Group (DIG) and Navy Intelligence and Information Warfare Branch (NI&IWB) within Navy Capability Division (NCD) have a requirement for Services to uplift and support ICT systems to enable capability to support provision of warfare analysis across Defence. The requirement for these Services is due to a synergised effort between initiatives led by ICD and NI&IW.  This effort will assist Navy and Joint Forces to enhance their user experience with both existing and developing capabilities related to ISR data transformation. The Services will support this synergised effort by facilitating further engagement between respective initiatives and stakeholders to elicit requirements from a programmatic view. Navy is delivering a multifaceted cyber hygiene program to enhance its cyber domain capabilities and resilience. ICD and NI&IW will assist Joint Forces to support integration of multiple ISR programs and projects into the Defence Secret Environment (DSE). Both ICD and NI&IW initiatives will establish strategic network connectivity with similar network design requirements. Alignment of these initiatives will achieve efficiencies in capabilities related to analysis and storage of Joint ISR, and Navy Acoustic and Cyber data, inclusive of Navy operational data. The Services must deliver the necessary ICT services, application development and systems integration to support capabilities that assists the Navy user community in warfare and cyber analysis. The Services must ensure the development of robust process workflows and applications that facilitate data uploads from various sources to standards-compliant object stores, and enable metadata tagging and provision of provenance information to operational data. |

| Commonwealth’s objectives For the NI&IW-led initiative, Navy faces increasing threat level and sophistication from cyber attacks against Information Technology (IT) and Operational Technology (OT) systems.  A successful attack could create extensive damage to IT and OT systems, which would then require recovery and/or restoration actions. In order to meet Chief of Navy requirements to Chief of Joint Capabilities (CJC) as the ADF Cyberworthiness authority, Navy is delivering a multifaceted cyber hygiene program to enhance its cyber domain capabilities and resilience, including development of a Cyber Terrain Library (CTL) prototype. Whilst this initiative has yielded promising results, it will not deliver a capability.  The CTL can record complete cyber topology, configurations and baselines of Navy IT and OT.  Data extracted from the CTL will allow cyber accreditation and certification as required by the Defence Security Principles Framework (DSPF). The introduction of a comprehensive cyber terrain management capability, using the CTL, will improve Navy’s Defensive Cyber Operations capability by: enabling holistic and complete understanding of own-force cyber systems; promoting recognition of security posture along with system vulnerabilities; reducing cyber incident response times; and identifying elements key to improving Navy’s defensive Cyberworthiness. The CTL will be installed on a number federated compute and storage nodes, as part of the Cyber Intelligence and Acoustic Network (CyIAN).  A new node will be established, whilst an existing (spare) node will be moved to a new geographical location. Federation of these nodes is to be enabled through enhancing connectivity between the sites, and upgrading facilities aspects such as power, heating/cooling, racking and the like. The overarching objective for CTL support services, is improvement to CTL data integrity & operation across the federated nodes, and assurance that CTL access & operation is available to users to a relatively high degree. The ICD-led initiative will contribute to the upgrade of Defence’s ICT systems to urgently deliver timely and effective ICT outcomes to support the ADF. The objective of the ICD-led initiative is to integrate data from ADF collection platforms to provide timely and secure access to data. This will support consistent situational awareness, targeting and decision advantage. The project will deliver the Joint ISR SECRET Environment, integrating components including data architecture and management, compute, networking and information services, technical support and the like. The project will scale to address growth in collection platforms and capabilities, and support emerging technologies. |

| Data Tagging FR-1.1: The application shall provide a user interface for tagging data with relevant metadata. FR-1.1.1: The tagging interface shall be accessible from the main dashboard. FR-1.1.2: Users shall be able to select predefined tags from a dropdown list. FR-1.2: The application shall allow users to create, edit, and delete tags. FR-1.2.1: Administrators shall have permissions to manage tag definitions. FR-1.2.2: The application shall log all changes to tags for auditing purposes. FR-1.3: The application shall support hierarchical tagging structures (parent-child relationships between tags). FR-1.3.1: Users shall be able to create nested tags up to three levels deep. FR-1.3.2: The interface shall visually indicate the hierarchy of tags. FR-1.4: The application shall enable users to apply multiple tags to a single data item. FR-1.4.1: The tagging interface shall allow multiple tag selection. FR-1.4.2: Tags applied to data items shall be searchable and filterable. Data Upload FR-2.1: The application shall provide a user interface for uploading data from various sources. FR-2.1.1: Users shall have the option to upload data via drag-and-drop or file selection. FR-2.1.2: The interface shall display a summary of the uploaded files before final submission. FR-2.2: The application shall suppot uploading data files of different formats, including CSV, JSON, XML, and others. FR-2.2.1: The application shall validate the format of uploaded files and provide error messages for unsupported formats. FR-2.2.2: The system shall support batch uploads for large data sets. FR-2.3: The applicaation shall allow users to upload data to S3-compliant object stores. FR-2.3.1: Users shall configure S3 bucket settings through the application. FR-2.3.2: The application shall confirm successful uploads and provide access URLs for uploaded objects. FR-2.4: The application shall provide progress indicators during data upload. FR-2.4.1: The progress bar shall display the percentage of data uploaded. FR-2.4.2: The application shall notify users of any interruptions during the upload process. FR-2.5: the application shall validate data files for compliace with predefined schemas before uploading. FR-2.5.1: The system shall provide predefined schemas for users to select. FR-2.5.2: Validation errors shall be displayed with detailed information for correction. Data Storage FR-3.1: The application shall store uploadd sata in S3-compliant object stores. FR-3.1.1: The application shall support versioning of stored objects. FR-3.1.2: The application shall ensure data is stored securely using encryption. FR-3.2: The application shall store metadata and tags in a relational database. FR -3.2.1: The database shall be optimised for fast retrieval of metadata and tags. FR-3.2.2: The system shall ensure referential integrity between data items and their tags. FR-3.3: The application shall ensure data integrity and prevent data loss during storage. FR-3.3.1: The system shall implement redundancy and backup mechanisms. FR-3.3.2: Regular integrity checks shall be performed to ensure data consistency. Data Management FR-4.1: The application shall provide functionalities for managing stored data, including viewing, editing, and deleting data records. FR-4.1.1: Users shall have a data management dashboard to perform these actions. FR-4.1.2: The system shall log all management actions for audit purposes. FR-4.2: The application shall support searching and filtering data based on tags and other metadata. FR-4.2.1: The search interface shall support complex queries, including AND/OR conditions. FR-4.2.2: Filter options shall include date ranges, file types, and custom metadata fields. FR-4.3: The application shall provide an audit trail for data management actions, including uploads, edits and deletions. FR-4.3.1: The audit trail shall be accessible only to authorized users. FR-4.3.2: Each audit entry shall include the action performed, the user who performed it, and the timestamp. User Management FR-5.1: The application shall support user authentication and authorisation. FR-5.1.1: The system shall support multiple authentication methods, including username/password and SSO. FR-5.1.2: The application shall enforce strong password policies. FR-5.2: The application shall provide role-based access control to restrict access to specific functionalities based on user roles. FR-5.2.1: Administrators shall define and assign roles with specific permissions. FR-5.2.2: The system shall prevent unauthorized access to restricted areas. FR-5.3: The application shall allow administrators to create, edit, and delete user accounts and roles. FR-5.3.1: The user management interface shall provide bulk user management capabilities. FR-5.3.2: The system shall notify users of changes to their account status via email. Data Integration FR-6.1: The application shall support integration with existing data sources. FR-6.1.1: The system shall provide connectors for popular data sources such as databases, file systems, and cloud services. FR-6.1.2: Integration settings shall be configurable through the application interface. FR-6.2: The application shall enable data transformation and mapping from various source formats to the application's format using Apache NiFi. FR-6.2.1: The system shall provide a graphical interface for designing data transformation workflows. FR-6.2.2: Users shall be able to save and reuse transformation templates. FR-6.3: The application shall provide real-time data synchronization with integrated sources. FR-6.3.1: The system shall support scheduled and on-demand synchronisation. FR-6.3.2: Notifications shall be sent to users upon successful or failed synchronisation. Reporting and Analytics FR-7.1: The application shall provide reporting functionalities to generate data insights and summaries. FR-7.1.1: Users shall be able to create custom reports using a report builder interface. FR-7.1.2: The application shall support automated report generation and scheduling. FR-7.2: The application shall support the export of reports in multiple formats, including PDF, Excel, and CSV. FR-7.2.1: Users shall be able to select the desired format during report export. FR-7.2.2: The system shall ensure the integrity and accuracy of exported data. FR-7.3: The application shall provide dashboards for visualizing key metrics and data trends. FR-7.3.1: Dashboards shall be customizable and support various visualization types (charts, graphs, tables). FR-7.3.2: Users shall be able to share dashboards with other users or groups. Usability and User Interface FR-8.1: The application shall provide a user-friendly and intuitive interface for all functionalities. FR-8.1.1: The interface shall be designed following usability best practices. FR-8.1.2: Tooltips and contextual help shall be provided throughout the application. FR-8.2: The application shall support responsive design to ensure usability across different devices and screen sizes. FR-8.2.1: The interface shall adapt to desktops, tablets, and smartphones. FR-8.2.2: The system shall maintain functionality and readability on all devices. FR-8.3: The application shall include help and documentation features to guide users through the application. FR-8.3.1: A searchable help section shall be available within the application. FR-8.3.2: The system shall provide video tutorials and FAQs for common tasks. Security FR-9.1: The application shall ensure secure data transmission using encryption protocols such as SSL/TLS. FR-9.1.1: All data transmitted between clients and servers shall be encrypted. FR-9.1.2: The system shall support the latest versions of SSL/TLS protocols. FR-9.2: The application shall implement data access controls to protect sensitive information. FR-9.3: The application shall provide logging and monitoring functionalities to track security events and potential breaches. Performance and Scalability FR-10.1: The application shall handle concurrent user access without performance degradation. FR-10.2: The application shall be scalable to handle increasing volumes of data and users. FR-10.3: The application shall provide performance monitoring tools to track system performance and identify bottlenecks. Compliance and Standards FR-11.1: The application shall comply with relevant industry standards and regulations for data security and privacy. FR-11.2: The application shall provide features to support data retention and deletion policies as required by regulations. Testinf and Quality Assurance FR-12.1: The application shall include unit tests, integration tests, and end-to-end tests to ensure functionality and reliability. FR-12.2: The application shall provide test environments to simulate real-world usage scenarios. FR-12.3: The application shall support continuous integration and deployment pipelines for automated testing and deployment. API Functional Requirements FR-13.1: The application shall provide a RESTful API to enable programmatic access to all major functionalities. FR-13.1.1: The API shall support CRUD (Create, Read, Update, Delete) operations for data records, tags, and metadata. FR-13.1.2: The API shall adhere to RESTful principles, including the use of standard HTTP methods (GET, POST, PUT, DELETE) and status codes. FR-13.1.3: The API endpoints shall be documented using a standard format such as OpenAPI (Swagger). FR-13.2: The API shall support authentication and authorisation mechanisms. FR-13.2.1: The API shall use OAuth 2.0 for authentication and authorization. FR-13.2.2: API keys and tokens shall be issued to authorised users and applications for accessing the API. FR-13.3: The API shall provide endpoints for data tagging operations. FR-13.3.1: Endpoints shall allow adding, updating, and deleting tags for specific data items. FR-13.3.2: The API shall support querying data items by tags and metadata. FR-13.4: The API shall allow uploading and retrieving data files. FR-13.4.1: The API shall support file upload operations to S3-compliant object stores. FR-13.4.2: Endpoints shall be available for retrieving file metadata and downloading files. FR-13.5: The API shall support integration with external systems. FR-13.5.1: Endpoints shall allow the synchronization of data with external sources. FR-13.5.2: The API shall provide webhooks for notifying external systems of significant events (e.g., data upload completion, tag changes) |
